# vista-meta

VistA metadata analytics sandbox — VEHU on YottaDB in Docker. Primary goal:
**PIKS classification** of every FileMan file (Patient, Institution, Knowledge,
System), then detailed schema modeling within each category.

## Quick reference

```bash
make build          # build Docker image (~20 min first time, layer 7 is expensive)
make run            # start container (services + background bake on first run)
make shell          # SSH into container as vehu
make mumps          # drop into YottaDB prompt inside container
make stop           # graceful shutdown
make help           # list all targets
```

## Where code runs

Code is **edited on the host**, **executed in the container**. The boundary matters.

| Location | Runs on | Contents |
|---|---|---|
| `docker/` | host (build context) | Dockerfile, entrypoint, configs |
| `vista/dev-r/` | container (bind mount) | Your M routines — shadows VEHU via $ZRO |
| `vista/scripts/` | container (bind mount) | Python + shell analysis scripts |
| `vista/export/` | container (bind mount) | Bake output, logs, sentinel JSON, research log |
| `vista/export/data-model/` | container (bind mount) | FileMan + PIKS extracted model (git-tracked) |
| `vista/export/code-model/` | container (bind mount) | Routines + packages + XINDEX extracted model (git-tracked) |
| `vista/export/RESEARCH.md` | container (bind mount) | Cross-cutting research log (RF-NNN, git-tracked) |
| `docs/` | host only | Spec, ADRs, build log, dependency manifest |
| `host/` | host only | Host-side Python venv for post-hoc analysis |
| `tests/smoke/` | host (against container) | Post-build verification |

## Do not edit

- `/opt/VistA-M/` inside the container — baked in the image, read-only at runtime
- Files under `vista/export/**/raw/` — regenerated by bake, gitignored
- `docs/adr/*.md` — immutable once accepted; supersede via new ADR

## Key architecture concepts

- **$ZRO layering**: `dev/r` (yours) -> `VistA-M/r` (VEHU) -> `$ydb_dist` (YDB libs).
  First match wins. Your routines shadow VEHU without mutating the image.
- **Hybrid persistence**: VEHU-M baked in image. Globals on named volume (survives
  `make rm`). Dev code on bind mounts (live-edits visible immediately).
- **ydb-run wrapper**: xinetd can't expand env vars, so `/usr/local/bin/ydb-run`
  sources the YDB env then exec's mumps. Both xinetd services use it.
- **Services bind 0.0.0.0 inside container**; Tailscale IP restriction is at the
  Docker `-p` level, not in service configs.

## PIKS classification (spec §11.5.2) — the primary analytical goal

Every FileMan file gets classified into one of four categories:
- **P**atient — clinical care data (protected, longitudinal, exchange-ready)
- **I**nstitution — facility, org, scope of care (comparative, admin)
- **K**nowledge — terminologies, templates, workflows (informatics, declarative)
- **S**ystem — config, operations, VistA internals (IT, DevOps)

Four orthogonal properties assigned alongside PIKS:
- **Volatility**: static / slow / dynamic / ephemeral
- **Sensitivity**: protected / operational / public
- **Portability**: universal / national / site-specific
- **Volume**: reference / moderate / high-volume

PIKS + properties come FIRST. They determine audience, security, storage,
and analytical approach before any detailed modeling.

## Analytical methodology (spec §11)

Analysis passes:
0a. **Global recon** — `VMCENSUS` Phase 1 scopes non-FileMan data (minutes)
0b. **Automated extraction** — `VM*` routines walk `^DD`/`^DIC` → complete TSVs
0c. **Full census** — `VMCENSUS` Phase 2 if recon shows significant non-FM data
1a. **Automated PIKS** — `VMPIKS` applies 52 heuristics (H-01–H-52) across 12 tiers
    + 6 non-FM heuristics (G-01–G-06) → 85-95% classified
1b. **Human triage** — 5-category triage (vestigial/package/mixed/cross-cutting/unresolved)
2. **Subdomain annotation** — semantic types, entity groupings within each PIKS

Four model layers: PIKS → Conceptual → Logical → Physical.
Every element traces down to `^DD` source. Coverage tracked in `coverage.json`.

## Conventions

- Spec cross-refs in code: `# Spec: docs/vista-meta-spec-v0.4.md § 5`
- ADR cross-refs in code: `# ADR-029: symlink farm for flat routine namespace`
- Build log cross-refs in spec: `[Errata: see BL-002]`
- Comment the **why**, not the what. If the code would look wrong to someone who
  doesn't know VistA, explain why it's right.
- TDD: tests before implementation. `make test` / `make check` before committing.
